# Brainstorm Summary

- Change: clean-hydrocore-baseline
- Date: 2026-07-17
- Language: zh-CN

## 已确认的事实

- 目标是把 HydroCore 整理成可靠的水处理系统二开基线。
- 范围覆盖根目录文档、`hydrocore-be` 后端、`hydrocore-fe` 前端、初始化 SQL、环境模板、Nacos/bootstrap 配置和本地 agent/IDE 配置残留。
- 非目标包括真实水处理工艺、业务表、监控大屏、控制算法、数字孪生、新业务 API 和大规模框架升级。
- OpenSpec 已定义新能力 `baseline-cleanup`，当前 design 阶段需要产出深度技术 Design Doc。

## 候选技术方案

### 方案 A：源入口优先清理（已确认）

先清理源码、SQL、Nacos/env 模板和用户可见文档；构建产物如 `target/`、`dist/` 不人工编辑，通过验证构建自然刷新。旧业务 schema 和配置采用删除或中性化，不改造成水处理业务模型。

优点：范围可控、审查清楚、最符合“二开基线”目标。风险：需要对少量旧术语做例外分类。

### 方案 B：全仓关键词强清理（候选）

对仓库做更激进的关键词替换和删除，尽量让旧业务词全局消失。

优点：表面残留最少。风险：容易误改生成产物、图标元数据、历史迁移文档和仍被通用 API 使用的命名，构建风险较高。

### 方案 C：预置水处理领域模型（不推荐）

把旧窑炉/预测/控制相关结构直接替换成水处理设备、工艺和监测模型。

优点：看起来更接近最终业务。风险：会在没有需求设计时固化错误领域假设，超出本 change 非目标。

## 确认的技术方案

用户已确认采用方案 A：源入口优先清理，按后端、前端、文档、验证四个工作流推进；保留通用底座 API；只删除或中性化旧业务语义，不创造新水处理领域能力。

初始化账号策略已确认采用方案 1：保留最小管理员/本地演示 seed 账号，但替换弱密码，并在文档中要求首次修改且不得用于生产。

## 关键取舍与风险

- 删除旧 seed 数据可能影响本地演示登录，因此需要确认初始化账号策略。
- 前后端系统配置枚举必须同步修改，否则配置页面和后端 API 可能不一致。
- `iconfont`、历史迁移文档和旧设计文档可能保留旧词，但必须作为例外记录。
- Maven 与前端构建是阶段验收硬门槛。

## 测试策略

- 后端：运行 `mvn -q -DskipTests compile` 和 `mvn -q test`。
- 前端：运行 `pnpm run build`。
- 人工/脚本检查：重新执行旧业务关键词盘点，确认只剩允许例外。
- 文档检查：根目录、后端、前端文档能说明启动、配置、验证和后续二开边界。
